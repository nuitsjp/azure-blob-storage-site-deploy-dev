# Design

## Background

### Why Blob Storage Instead of SWA

There are two reasons for not adopting Azure Static Web Apps (SWA).

1. **Hosting multiple environments on a single resource**: SWA is designed to serve a single site. It is not suited for simultaneously publishing multiple persistent environments (such as main and develop, which are not staging environments) within a single Azure resource. The static website feature of Blob Storage enables multi-site hosting via subdirectories.
2. **Access control via Private Endpoint**: Private Endpoints cannot be applied to SWA's static website endpoints. With Blob Storage, a Private Endpoint can be attached to restrict access exclusively to the internal corporate network.

### Scope

- Target content is limited to HTML files generated by static site generators
- SPA routing (such as React) is not supported

## Design Principles

### Directory Structure

Prefixed subdirectories are created within the `$web` container to enable multi-site hosting.

```
$web/
├── index.html          ← Root (branch listing, etc. — optional)
├── main/               ← Released documentation (persistent)
│   ├── index.html
│   └── ...
├── develop/            ← In-development documentation (persistent)
│   ├── index.html
│   └── ...
├── pr-42/              ← Staging for PR #42 (temporary)
│   ├── index.html
│   └── ...
└── pr-57/              ← Staging for PR #57 (temporary)
    ├── index.html
    └── ...
```

Example access URLs:
- `https://<account>.z11.web.core.windows.net/main/`
- `https://<account>.z11.web.core.windows.net/pr-42/`

### Prefix Naming Conventions

| Type | Prefix | Example | Lifecycle |
|---|---|---|---|
| Persistent branch | Branch name as-is | `main`, `develop`, etc. | Overwritten on each push. Target branches are declared in the caller workflow |
| PR staging | `pr-<PR number>` | `pr-42` | Created on PR open, deleted on PR close |

### Why PR Number-Based Prefixes

Using branch names directly as directory names has the following issues.

1. **Japanese branch name issue**: If a branch name is entirely in Japanese, sanitizing to ASCII results in an empty string, causing a breakdown. Inconsistencies in URL encoding and unstable behavior in Azure CLI/shell are also concerns.
2. **Slash issue**: Branch names like `feature/add-docs` have `/` interpreted as a directory separator, causing unintended nesting.
3. **Distinguishing from persistent branches**: Deletion targets on PR close must be clearly distinguishable. With branch name-based prefixes, there is a risk of accidentally deleting persistent branch directories (such as main).

PR numbers (`github.event.pull_request.number`) are unique, immutable integers that avoid all of these issues. Persistent branches are operationally restricted to ASCII, so using the branch name directly is safe.

### Staging Environment Lifecycle

| GitHub Event | Trigger Condition | Action |
|---|---|---|
| `push` | Persistent branches declared in caller workflow | Deploy to the branch name directory (overwrite) |
| `pull_request` opened | On PR creation | Deploy to `pr-<number>` directory |
| `pull_request` synchronize | On push to PR branch | Update `pr-<number>` directory (overwrite) |
| `pull_request` closed | On PR close (regardless of merge) | Delete `pr-<number>` directory |

### Concurrent Deployment Control

Multiple PRs deploying simultaneously do not conflict because their prefixes are different. Consecutive pushes within the same PR are controlled via GitHub Actions concurrency groups.

The deployment target prefix (PR number or branch name) is used as the concurrency group key, with `cancel-in-progress: true` to cancel preceding jobs.

## Action Interface

### Separation of Responsibilities

Similar to SWA's GitHub Actions, this action is responsible **only for uploading and deleting pre-built artifacts**. Building documentation is the caller workflow's responsibility.

**Why this separation**: There are many static site generators (Sphinx, Hugo, etc.), and build procedures differ per project. By keeping the action independent of the build process, reusability is ensured.

### Inputs

| input | Description | Required |
|---|---|---|
| `storage_account` | Azure Storage account name | Yes |
| `source_dir` | Path to the directory to upload (only when `action=deploy`) | Conditional |
| `branch_name` | Branch name. Used as prefix when `pull_request_number` is not specified | Conditional |
| `pull_request_number` | PR number. When specified, `pr-<number>` is used as the prefix | Conditional |
| `action` | `deploy` (upload) or `cleanup` (delete) | Yes |

> Either `branch_name` or `pull_request_number` is required. If `pull_request_number` is specified, it takes priority.

### Prefix Resolution Logic

The action internally determines the deployment target prefix using the following priority. The caller does not need to assemble the prefix or implement branching logic.

1. If `pull_request_number` is specified -> `pr-<number>`
2. If `pull_request_number` is not specified -> use `branch_name` as-is
3. If neither is specified -> validation error

### Caller Workflow Processing Flow

**Deploy job** (executed on events other than PR close):

1. Check out the repository
2. Build documentation (project-specific procedure)
3. Authenticate with Azure via `azure/login`
4. Call this action with `action=deploy`, passing `branch_name` and `pull_request_number` (prefix is resolved automatically within the action)

**Cleanup job** (executed only on PR close):

1. Authenticate with Azure via `azure/login`
2. Call this action with `action=cleanup`, passing `pull_request_number`

### Azure Authentication

Authentication from GitHub Actions assumes the OIDC (Federated Credentials) method. The action's prerequisite is that the caller has already authenticated via `azure/login`.

**Why OIDC**: Compared to storage account keys and SAS tokens, OIDC eliminates the need for secret management and rotation, and is the safest and most recommended method for GitHub Actions integration.

## Technical Constraints and Mitigations

### Static Website Index Document

By setting "Index document" to `index.html` in the Azure Blob Storage static website configuration, accessing `/main/` returns `/main/index.html`. This works per subdirectory, so there is no issue.

### Trailing Slash Issue

Azure Blob Storage's static website does not automatically redirect `/pr-42` to `/pr-42/`. When accessed without a trailing slash, relative paths within the page (CSS/JS/images) resolve against the root, causing broken layouts.

This behavior is an Azure specification and cannot be controlled by the static website feature alone (an improvement request has been raised on Tech Community).

**Mitigation**: As an operational rule, URLs distributed via PR comments must always include a trailing `/`. In the future, if a CDN/Front Door is placed in front, redirect rules can handle this.

### Error Document

Only one 404 page can be set per storage account. Custom 404 pages per branch are not possible. However, since static site generator output consists of individual HTML files for every page, 404 errors are rare and this is not a practical issue.

### Storage Capacity and Billing

Since the entire site is copied per PR, storage usage increases proportionally to the number of simultaneously open PRs. Considering the size of documentation sites (typically tens of MB), the cost impact is negligible. If needed, a cleanup job (automatic deletion of long-abandoned PR staging environments) can be added.

### File Synchronization During Deployment

`az storage blob upload-batch` only uploads new and updated files and does not automatically delete files from the Blob side that were deleted on the source side. Without mitigation, old files would persist on Blob storage when documentation pages are deleted or renamed.

**Mitigation**: When the `deploy` action runs, it first deletes all files under the target prefix via `delete-batch`, then uploads via `upload-batch`. This guarantees complete synchronization between the source and Blob storage.

### Access Scope

Azure Blob Storage's static website feature fundamentally delivers content via anonymous GET and does not support access control via Entra ID.

In this design, a Private Endpoint is attached to the storage account, and access is permitted only through that endpoint. This avoids viewer authentication and account management concerns while achieving internal-only publication.

## Roadmap

1. Implement the Composite Action (`action.yml`)
   - `deploy`: Delete existing files with `az storage blob delete-batch`, then upload with `upload-batch`
   - `cleanup`: Delete with `az storage blob delete-batch`
2. Create caller workflow templates
3. Auto-post staging URL to PR comments (optional)
4. Auto-generate root page (branch listing) (optional)
